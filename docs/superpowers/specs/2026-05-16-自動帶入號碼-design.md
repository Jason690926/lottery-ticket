# 自動帶入最新/上一期號碼 設計

**日期：** 2026-05-16
**狀態：** Approved（待 implementation plan）
**範圍：** 威力彩 + 大樂透兩個分頁，自動把最新一期號碼帶入第一選號區塊、上一期帶入第二選號區塊，並自動產生分析結果

---

## 1. 目的

每次資料更新後，使用者打開工具不必手動輸入號碼，系統自動：

- **第一選號區塊（lag-1，本期號碼）** ← 最新一期開獎號碼（6 主號 + 第二區/特別號）
- **第二選號區塊（lag-2，上一期號碼）** ← 上一期開獎號碼（6 主號 + 第二區/特別號）

號碼帶入後分析結果自動呈現（現有 UI 本就「widget 有值即渲染」，「開始分析」按鈕只控制提示訊息，不控制分析本身）。

## 2. 範圍

**包含：**
- `query_engine.py` 新增 `latest_two_draws(cfg)`
- `ui_shared.py` 新增 `seed_inputs(cfg, key_prefix)`
- `ui_powerball.render()` / `ui_lotto649.render()` 各插入一次呼叫 + 重設按鈕 + caption
- AppTest 自動化驗證

**不包含：**
- 不改 `zone1_analysis` / `zone1_dual_combined` / `zone2_analysis` / `_get_draws` / 資料層 / `app.py` 入口
- 不改分析演算法、不影響威力彩既有分析行為
- 不做「跨 session 記住使用者偏好」（session 內有效即可）

## 3. 行為規格（已與使用者確認）

| 決策 | 結論 |
|------|------|
| 預填性質 | **只當預設**：開啟 / 資料出現新一期時帶入；使用者手改後以他為準，不再覆蓋 |
| 新一期到達 | 重新種子為新期（舊選號被覆蓋，因為「新期 = 新預設」，舊期已過時） |
| 重設機制 | 每個彩種 tab 加「↻ 帶回最新期」按鈕，隨時把 4 個輸入梳回最新/上一期 |
| 自動產生結果 | 不需額外動作；widget 被種子化後分析即自動渲染 |

## 4. 核心技術挑戰

Streamlit 每次互動都重跑整個 script。要做到「只當預設、手改後不覆蓋、新一期才重帶」，
不能每次重跑都寫死預填，必須用 `st.session_state` + 一個「已帶入的 draw_id」哨兵值，
僅在「沒種過」或「最新 draw_id 改變」時才寫入 widget 的 session_state。

## 5. 介面設計

### 5.1 `query_engine.latest_two_draws`

```python
def latest_two_draws(cfg: LotteryConfig = POWERBALL) -> tuple[dict | None, dict | None, str]:
    """
    回傳 (最新期, 上一期, 最新 draw_id)。
    每個 dict: {"draw_id": str, "draw_date": str, "z1": [n1..n6], "z2": int}
    - 無資料：           (None, None, "")
    - 只有 1 期：         (最新, None, latest_id)
    - 2 期以上：          (最新, 上一期, latest_id)
    沿用 _get_draws(2, cfg)（已依 id 升冪；最後一列=最新、倒數第二=上一期）。
    """
```

DB 存取集中在 `query_engine`，與現有 `_get_draws` 架構一致。

### 5.2 `ui_shared.seed_inputs`

```python
def seed_inputs(cfg, key_prefix: str) -> tuple[str, str, str, str]:
    """
    在 widget 建立『之前』呼叫。依 draw_id 哨兵決定是否種子化 4 個 widget。
    回傳 (latest_id, latest_date, prev_id, prev_date) 供 caption 顯示；
    無資料時回傳 ("", "", "", "")。
    """
```

session_state key（每個彩種獨立，前綴 `pb` / `lt`）：

- 既有 widget key：`{p}_b1_z1`、`{p}_b1_z2`、`{p}_b2_z1`、`{p}_b2_z2`
- 新哨兵 key：`{p}_seeded_id`

邏輯：

```
latest, prev, latest_id = latest_two_draws(cfg)
if latest is None:
    return ("", "", "", "")                       # 空表防呆
if st.session_state.get(f"{p}_seeded_id") == latest_id:
    # 已是最新期且種過 → no-op，保留使用者手改值
    pass
else:
    st.session_state[f"{p}_b1_z1"] = latest["z1"]
    st.session_state[f"{p}_b1_z2"] = latest["z2"]
    st.session_state[f"{p}_b2_z1"] = prev["z1"] if prev else []
    st.session_state[f"{p}_b2_z2"] = prev["z2"] if prev else None
    st.session_state[f"{p}_seeded_id"] = latest_id
return (latest_id, latest["draw_date"],
        prev["draw_id"] if prev else "",
        prev["draw_date"] if prev else "")
```

### 5.3 widget 不需改寫

現有 4 個 widget 本來就只給 `key=`、未傳 `default=`/`index=`，
故先寫 `session_state` 再建 widget 不會觸發 Streamlit「default 與 session_state 衝突」警告。
唯一改動：在兩個 input 區塊建立『之前』插入 `seed_inputs(cfg, "pb"/"lt")` 呼叫。

未種子化時（空表）：multiselect 自然為 `[]`、selectbox 自然為第一個選項 `None`（「── 請選擇 ──」），行為與現況一致。

### 5.4 重設按鈕

- 位置：input 區下方，與既有「🔍 開始分析」同一列旁。
- callback：`st.session_state.pop(f"{p}_seeded_id", None)` 然後 `st.rerun()`。
  下一輪 `seed_inputs` 因哨兵消失 → 重新種子（覆蓋手改值）。不需手動清 4 個 widget key。

### 5.5 Caption

input 區下方 `st.caption`：

- 正常：`🔄 已自動帶入最新一期 {latest_id}（{latest_date}）為第一區塊、上一期 {prev_id}（{prev_date}）為第二區塊。手動修改後可按「↻ 帶回最新期」復原。`
- 只有 1 期：省略「上一期…」子句。
- 空表：`⚠️ 尚無開獎資料，未自動帶入。`

## 6. 資料流時序

1. app 啟動 → `_auto_update_db()` 補最新開獎進 DB（已存在功能）
2. tab `render()` → **先**呼叫 `seed_inputs()`（讀 DB 最新兩期、視哨兵決定是否寫 session_state）
3. **後**建立 4 個 input widget（值由 session_state 決定）
4. `any_z1 / any_z2` 因 widget 有值 → 分析結果自動渲染

## 7. 邊界情況

| 情況 | 行為 |
|------|------|
| DB 該彩種 0 期 | 不種子、widget 空白、caption 顯示「尚無開獎資料」 |
| 只有 1 期 | b1 帶最新、b2 留空（`[]` / `None`） |
| 使用者手改後重跑（同期） | 哨兵相符 → no-op，保留手改值 |
| 開著 app 跨到新一期 | 哨兵不符 → 重新種子為新期 |
| 第一區帶 6 主號 | 開獎本就 6 號，與 `max_selections=6` 一致 |
| 威力彩第二區 1–8 / 大樂透特別號 1–49 | `latest["z2"]` 為 `n_zone2` 整數，必在對應 selectbox options 內 |

## 8. 測試策略

既有專案無 pytest，沿用 `streamlit.testing.v1.AppTest` + `python -c` ad-hoc 驗證：

1. `latest_two_draws`：威力彩/大樂透各比對 DB 最後兩列，欄位與 draw_id 正確
2. AppTest 全新啟動：4 個 widget 的 session_state == DB 最新/上一期
3. 模擬手改 + 重跑（哨兵相符）：值保留不被覆蓋
4. 模擬哨兵不符（新一期）：重新種子
5. 重設：pop 哨兵 + rerun → 重新種子
6. 威力彩 + 大樂透各驗一次；AppTest 整體無 exception

## 9. 風險與權衡

| 風險 | 緩解 |
|------|------|
| Streamlit 版本對「widget 已實例化後改 session_state」限制 | `seed_inputs` 一律在 widget 建立前呼叫；重設走 `st.rerun()` 不在事後改值 |
| 種子化把使用者珍貴的手改值蓋掉 | 哨兵機制：同期只種一次；新期才覆蓋（符合需求且已與使用者確認） |
| `_get_draws` 依 `id` 排序，回灌/增量混合後順序 | 回灌按月升冪、增量在後，`id` AUTOINCREMENT 即時間序；與分析引擎同一套排序，一致 |
| caption 日期格式 | 直接用 DB `draw_date`（已是 `YYYY/MM/DD`），不另格式化 |
