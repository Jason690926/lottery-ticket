# 大樂透分頁設計

**日期：** 2026-05-15
**狀態：** Approved（待 implementation plan）
**範圍：** 在現有威力彩 Streamlit 工具上新增大樂透分頁，採同樣的條件頻率分析邏輯

---

## 1. 目的

把現有的「威力彩條件篩選系統」擴充為支援多彩種，第一個新增彩種為大樂透（1–49 選 6 + 特別號 1 個）。
未來新增任何彩種（雙贏彩、今彩 539 等）都應能透過「再寫一個 LotteryConfig」完成，不必動引擎邏輯。

## 2. 範圍

**包含：**
- SQLite 新表 `draws_lotto649`
- 大樂透歷史資料回灌（2014-01 起）+ 增量更新腳本
- `query_engine.py` 重構為參數化（接 `LotteryConfig`）
- `app.py` 拆檔 + 加頂部 tab 切換
- 大樂透 UI（第一區 + 第二區/特別號分析，鏡射威力彩結構）

**不包含：**
- 大樂透不切 holdout（這次純當選號工具，不做研究週期）
- 不做精確版特別號邏輯（不排除「特別號 = 主號之一」的不可能組合，採簡化版）
- 不重做統計顯著性檢定（Phase 1 結論：與 i.i.d. 無法區分；本工具僅作有系統的選號參考）

## 3. 路線決定

採「**參數化共用引擎**」：`query_engine.py` 接 `LotteryConfig`，威力彩/大樂透各傳一份設定。

否決方案：
- **複製 `query_engine_lotto.py`**：兩份程式以後會 drift；改演算法要改兩邊
- **OOP class 階層**：目前只有 2 個彩種，過度抽象

## 4. 資料層

### 4.1 SQLite schema（新增表）

```sql
CREATE TABLE draws_lotto649 (
    draw_id     TEXT PRIMARY KEY,    -- 期別
    draw_date   TEXT NOT NULL,       -- 'YYYY/MM/DD'
    n1 INT, n2 INT, n3 INT, n4 INT, n5 INT, n6 INT,
    n_zone2     INT,                 -- 大樂透這欄存「特別號」（語意延伸，避免 query 引擎多一個 col 參數）
    total_sales INT DEFAULT 0,
    split TEXT DEFAULT 'live'        -- 'train' / 'live'（不切 holdout）
);
CREATE INDEX idx_lotto649_date ON draws_lotto649(draw_date);
```

威力彩 `draws` 表完全不動。

### 4.2 歷史回灌（一次性）

新檔 `backfill_lotto649.py`：
- 從台彩 Lotto649 API 抓 **2014-01 起** 至今所有期數
- 全部標 `split='train'`
- 跑完即可保留或刪除

### 4.3 增量更新

新檔 `update_db_lotto649.py`：
- 鏡像 `update_db.py`，改 API endpoint
- 每次抓最近 3 個月，`INSERT OR IGNORE` 補入新期
- 新期標 `split='live'`

### 4.4 啟動時自動更新（`app.py`）

```python
def _auto_update_db():
    for mod in ("update_db", "update_db_lotto649"):
        try: __import__(mod).main()
        except Exception: pass
```

## 5. 引擎層（`query_engine.py`）

### 5.1 LotteryConfig

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class LotteryConfig:
    name: str               # 'powerball' / 'lotto649'
    table: str              # 'draws' / 'draws_lotto649'
    z1_max: int             # 38 / 49
    z1_pick: int            # 6 / 6
    z1_baseline_pct: float  # 15.8 / 12.2  ← 顯示在 metric 與基準線
    has_zone2: bool         # True / True
    z2_max: int             # 8 / 49
    z2_baseline_pct: float  # 12.5 / 2.04

POWERBALL = LotteryConfig("powerball", "draws", 38, 6, 15.8, True, 8, 12.5)
LOTTO649  = LotteryConfig("lotto649", "draws_lotto649", 49, 6, 12.2, True, 49, 2.04)
```

`z1_baseline_pct` 計算：6 / z1_max × 100，但寫死避免每次重算（也方便調整）。

`z2_baseline_pct` 推導：
- 威力彩第二區：1/8 = 12.5%
- 大樂透特別號：P(x = 特別號) = P(x ∉ 主6) × P(x 被抽中為特別號 | x ∉ 主6) = 43/49 × 1/43 = **1/49 ≈ 2.04%**

### 5.2 函式簽名

加 `cfg: LotteryConfig = POWERBALL` 預設參數，現有 `app.py` 既有 import 完全不必改、行為不變。

```python
def zone1_analysis(condition_nums, rolling=ROLLING, lag=1, cfg=POWERBALL): ...
def zone1_dual_combined(b1, b2, rolling=ROLLING, w1=0.5, w2=0.5, cfg=POWERBALL): ...
def zone2_analysis(cond, rolling=ROLLING, cfg=POWERBALL): ...
def _get_draws(n, cfg=POWERBALL): ...
```

### 5.3 內部變動點（最小修改）

- `_get_draws`：SQL 改 `f"SELECT ... FROM {cfg.table}"`
- `zone1_analysis`：所有寫死的 `38` → `cfg.z1_max`、`range(1, 39)` → `range(1, cfg.z1_max+1)`、numpy shape `(T, 38)` → `(T, cfg.z1_max)`
- `zone1_dual_combined`：跟著改 38 → `cfg.z1_max`、`range(1, 39)` → `range(1, cfg.z1_max+1)`
- `zone2_analysis`：`range(1, 9)` → `range(1, cfg.z2_max+1)`

### 5.4 重構安全網（Sanity Check）

重構前先用幾組已知條件跑一次威力彩、把結果 `pickle` 存檔。重構後跑同樣輸入，`assert df_old.equals(df_new)`。確認後刪掉 pickle。不寫成正式 test。

## 6. UI 層

### 6.1 檔案組織

把 559 行的 `app.py` 拆成：

```
app.py                # 入口：CSS、header、auto_update、頂部 tab 切換
ui_shared.py          # 共用：_plotly_base()、PALETTE_B1/B2、_render_block_table/bar/line/section
ui_powerball.py       # render_powerball_section()
ui_lotto649.py        # render_lotto649_section()
query_engine.py       # 共用引擎（已參數化）
update_db.py          # 威力彩
update_db_lotto649.py # 大樂透
backfill_lotto649.py  # 一次性歷史回灌
```

理由：再塞大樂透 `app.py` 會爆炸；各彩種一個 UI 檔，互不干擾。

### 6.2 頂層 tab 切換

```python
tab_pb, tab_lt = st.tabs(["🎯 威力彩", "🍀 大樂透"])
with tab_pb: render_powerball_section()
with tab_lt: render_lotto649_section()
```

### 6.3 大樂透 UI vs 威力彩 UI 差異

| 項目 | 威力彩 | 大樂透 |
|------|--------|--------|
| 第一區號碼範圍 | 1–38（max 6） | 1–49（max 6） |
| 第一區理論基準 | 15.8% | 12.2% |
| top_n slider | 5–38（預設 15） | 5–49（預設 18） |
| 第二區輸入 | selectbox 1–8 | selectbox 1–49（49 個選項，可滾動） |
| 第二區標籤 | 「第二區號碼（1–8）」 | 「特別號（1–49）」 |
| 第二區基準 | 12.5% | 2.04% |
| 第二區 x 軸 | 8 個 bar | 49 個 bar（密集但可讀） |
| 配色 | 藍/橘 B1/B2 | 同樣藍/橘 B1/B2 |

**其他都一樣**：兩個輸入區塊（B1 本期 + B2 上一期）、lag-1 / lag-2 / 加權合併三個 tab、所有圖表類型、深色主題、warn-card 統計說明。

### 6.4 副標題（資料最新至）

各 tab 自己渲染副標題（`資料最新至 YYYY/MM/DD`），分別讀自己的表。主標題仍共用。

### 6.5 特別號輸入約束

不限制（即使選的特別號跟主號重複，UI 仍允許）。原因：嚴格防呆 UI 複雜度高，分析意義有限。在輸入區下方加一行小字提示：「特別號分析採簡化邏輯，不排除與主號重複的不可能組合」。

## 7. 風險與權衡

| 風險 | 緩解 |
|------|------|
| Lotto649 API endpoint / 欄位名未驗證 | 寫 `update_db_lotto649.py` 第一步打一次 API、印 response、確認欄位後再對 |
| 重構 `query_engine.py` 改壞威力彩 | 5.4 的 pickle sanity check |
| 拆檔後 `app.py` 還會用到全域 helper | 全部移到 `ui_shared.py`，兩個 UI 檔 import |
| 49 個 bar 的圖太密 | 先做出來看，太密的話 top_n 預設提高到 20、或加 x 軸標籤旋轉 |
| 簡化版特別號邏輯不夠精確 | 文件 + UI 提示揭露此簡化；未來可加 `exact_special` 模式 |

## 8. 驗收標準

- [ ] 威力彩 tab 行為與重構前完全一致（pickle sanity check 通過）
- [ ] 大樂透 tab 第一區輸入 1–49、選號上限 6、可跑出 lag-1 / lag-2 / 加權合併三個分析
- [ ] 大樂透 tab 特別號輸入 1–49、可跑出單區塊 + 合併比較分析
- [ ] `lottery.db` 中 `draws_lotto649` 表筆數 ≥ 1,200（2014-01 起的歷史 + 最新增量）
- [ ] app 啟動時兩個彩種都會 silent auto-update
- [ ] 兩個 tab 副標題各自顯示自己的「資料最新至」
